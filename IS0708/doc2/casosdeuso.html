Caso de uso 01: Autentificar al usuario

Objetivo en contexto

Acceder al sistema.

Precondiciones

Debe ser posible establecer la conexión con la base de datos.

Postcondiciones si éxito

Se accede al sistema.

Postcondiciones si fallo

Se piden de nuevo los datos.

Actores

Usuario.

Secuencia normal

1- Introducir número de usuario.

2- Introducir contraseña.

3- Confirmar.

4- Acceder al sistema.

Secuencias alternativas

3a- Si el usuario no está en el sistema da un error y se da la opción a introducir de nuevo los datos.

3b- Si la contraseña no corresponde al usuario da un error y se da la opción a introducir de nuevo los datos.

Caso de uso 22: Modificar los datos de un empleado

Objetivo en contexto

Tener el sistema actualizado con los nuevos datos del empleado.

Precondiciones

El jefe de departamento ha accedido correctamente al sistema y ya está en su menú de opciones.

Postcondiciones si éxito

Los datos del empleado quedan actualizados.

Postcondiciones si fallo

Actores

Jefe de departamento.

Secuencia normal

1- Elegir opción "Modificar datos".

2- Introducir los datos a modificar en lugar de los antiguos.

3- Confirmar.

4- Se actualizan los datos del empleado en la base de datos.

5- Se modifica el cuadrante si es necesario (CU-27) y se notifica al jefe de departamento.

Secuencias alternativas

2a- El empleado no está en el sistema. Comprobar que se ha escrito bien el identicador.

2b- Si los datos está en formato incorrecto la aplicación permite corregir los datos erróneos marcando los campos incorrectos de alguna forma (vuelta a 2).

2c- Se accede a algún dato que no se puede modificar. Modificar los datos permitidos o cancelar la operación.

Secuencias alternativas

2a- Si los datos está en formato incorrecto da error.

5a- Si el contrato ya existe da error.

5b- Si hay un error en la conexión se muestra por pantalla.

Historial de cambios

Diciembre 2007

El caso de uso 32 (Configurar el idioma) desaparece puesto que el idioma se elige al dar de alta al usuario.

El caso de uso 31 (Configurar la aplicación) desaparece ya que no hay ninguna opción de configuración de la aplicación.

Se añade el caso de uso 5 (Responder mensaje) cambiando la numeración de los posteriores casos de uso.

Se añade el caso de uso 32 (Imprimir).

Se añade al caso de uso 33 (Solicitar ayuda).



Enero 2008

En los casos de uso 07 y 08 el actor debería ser el empleado ya que se lo pide al jefe de departamento.

En los casos de uso 10 y 11 los actores serían el jefe de departamento y el gerente.

En el caso de uso 16, así como el gerente es el único que puede crear departamentos, es él el único que los puede eliminar.

En los casos de uso 21, 22 y 26 el caso de uso de modificar cuadrante al que se hace referencia es el 27 y no el 21.

En los casos de uso 23, 24 y 25, al tocar el cuadrante puede ser necesario cambiar el cuadrante.

En el caso de uso 27 cuando se hace referencia al caso de uso 23 en realidad es el caso de uso 29, y cuando se hace referencia al caso de uso 22 es el caso de uso 28.

Se ha extendido el caso de uso 01 para cuando se dispone de lector de códigos de barras añadiendo el caso de uso 34 (Autentificar al usuario con lector de códigos de barras).



Marzo 2008

En los casos de uso 23, 24 y 25 el actor es el jefe de departamento y cambia las secuencias normal y alternativa.

Se modifica el caso de uso 27, que pasa a llamarse Generar cuadrante.

Se modifica el caso de uso 28 ya que ahora no es posible que no se genere un cuadrante. Ahora el caso de uso 28, Sugerir más personal según el cuadrante generado, se refiere al caso en que un cuadrante se genera con falta de personal.

Se añade el caso de uso 35 (Crear turno).

Se añade el caso de uso 36 (Modificar turno).

Se añade el caso de uso 37 (Eliminar turno).

Se añade el caso de uso 38 (Asignar turno inicial).

Se añade el caso de uso 39 (Configurar el patrón de un contrato).
