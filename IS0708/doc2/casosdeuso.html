Caso de uso 01: Autentificar

Objetivo en contexto

Acceder al sistema.

Precondiciones

Debe ser posible establecer la conexión con la base de datos.

Postcondiciones si éxito

Se accede al sistema.

Postcondiciones si fallo

Se piden de nuevo los datos.

Actores

Usuario y base de datos.

Secuencia normal

1- Introducir número de usuario.

2- Introducir contraseña.

3- Aceptar.

4- Acceder al sistema.

Secuencias alternativas

3a- Si el usuario no está en el sistema da un error y se da la opción a introducir de nuevo los datos.

3b- Si la contraseña no corresponde al usuario da un error y se da la opción a introducir de nuevo los datos.




Caso de uso 02: Dar de alta a un empleado

Objetivo en contexto

Introducir un nuevo empleado en la base de datos.

Precondiciones

El jefe de departamento ha accedido correctamente al sistema y ya está en su menú de opciones.

Postcondiciones si éxito

El empleado queda registrado.

Postcondiciones si fallo

Mensaje de error.

Actores

Jefe de departamento y base de datos.

Secuencia normal

1- elegir opción dar de alta.

2- Introducir datos.

3- Confirmar.

4- Se añade el nuevo empleado a la base de datos.

Secuencias alternativas

2a- El empleado ya está en el sistema. No es necesario darlo de alta.

2b- Datos con formato incorrecto. La aplicación permite corregir los datos erróneos marcando los campos incorrectos de alguna forma (vuelta a 2).




Caso de uso 03: Dar de baja a un empleado

Objetivo en contexto

Eliminar un empleado de la base de datos.

Precondiciones

El jefe de departamento ha accedido correctamente al sistema y ya está en su menú de opciones.

Postcondiciones si éxito

El empleado queda eliminado del sistema.

Postcondiciones si fallo

Actores

Jefe de departamento y base de datos.

Secuencia normal

1- Elegir opción dar de baja.

2- Introducir identificador del empleado a eliminar.

3- Confirmar.

4- Se borra el nuevo empleado de la base de datos.

5- Si es necesario, se modifica el cuadrante (CU-15).

Secuencias alternativas

2a- El empleado no está en el sistema. No es necesario darle de baja.

2b- Identificafor incorrecto. La aplicación solicita vbolver a introducir el identificador.




Caso de uso 04: Modificar los datos de un empleado

Objetivo en contexto

Tener el sistema actualizado con los nuevos datos del empleado.

Precondiciones

El jefe de departamento ha accedido correctamente al sistema y ya está en su menú de opciones.

Postcondiciones si éxito

Los datos del empleado quedan actualizados.

Postcondiciones si fallo

Actores

Jefe de departamento y base de datos.

Secuencia normal

1- Elegir opción de modificar.

2- Introducir los datos a modificar en lugar de los antiguos.

3- Confirmar.

4- Se actualizan los datos del empleado en la base de datos.

5- Se modifica el cuadrante si es necesario (CU-15).

Secuencias alternativas

2a- El empleado no está en el sistema. Comprobar que se ha escrito bien el identicador.

2b- Datos con formato incorrecto. La aplicación permite corregir los datos erróneos marcando los campos incorrectos de alguna forma (vuelta a 2).

2c- Se accede a algún dato que no se puede modificar. Modificar los datos permitidos o cancelar la operación.

Caso de uso 15: Crear/modificar cuadrantes

Objetivo en contexto

Creación/modificación del cuadrante (considerando la creación como la modificación de un cuadrante vacío).

Precondiciones

Estar identificado, y debe ser posible establecer la conexión con la base de datos.

Postcondiciones si éxito

Cuadrante correcto.

Postcondiciones si fallo

Ninguna modificación.

Actores

Jefe de departamento, gerente y base de datos.

Secuencia normal

1- Seleccionar opción modificar cuadrante.

2- Introducir datos si es necesario.

3- La aplicación propone un cuadrante alternativo.

4- Confirmación para creación de cuadrante.

5- Notificación del cuadrante a los usuarios afectados (CU-16).

Secuencias alternativas

4a.1a- El usuario rechaza la propuesta, y la apliación vuelve al punto 3.

4a.1b- La aplicación no puede generar otro cuadrante.

4a.1b.1a- El usuario vuelve al principio para revisar las propuestas (vuelve al punto 3).

4a.1b.1b- El usuario cambia las condiciones iniciales para intentar generar otro cuadrante.




Caso de uso 16: Enviar mensajes de forma automática

Objetivo en contexto

Generar avisos de forma automática tras haber realizado modificaciones en el cuadrante.

Estar identificado, y debe ser posible establecer la conexión con la base de datos.

Haber realizado alguna modificación.

Tener una lista de usuarios a los que hay que avisar.

Tener un mensaje de aviso.

Postcondiciones si éxito

Entregar los avisos correspondientes a los usuarios pertinentes.

Postcondiciones si fallo

Pantalla de error.

Actores

Base de datos.

Secuencia normal

1- Enviar mensajes de forma automática.

2- Mostrar un mensaje indicando que el envío de mensajes se ha producido con éxito.

Secuencias alternativas

2a- Si hay un fallo, mostrar un mensaje indicando quién falta por avisar y por qué se ha producido el error.

2b- Si hay un error en la conexión se muestra por pantalla.
